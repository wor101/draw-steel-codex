What's changing in the editor

These are the visible differences between the classic Triggered Ability editor and the new design. Everything here is pure UI/UX; the data model and engine are unchanged.

  • Preview structure. Single flat description textarea on the right → three stacked panes (card preview, mechanical view, Test Trigger strip). new
  • Navigation. Single long scrollable form → four sections in a left nav (Trigger, Response, Effects, Display). Sections never overlap; one active at a time. new
  • Validation feedback. Classic: silent, author runs the ability in-game and checks. New: live chips on the mechanical view (valid / unknown symbol / empty / subject mismatch) plus a rollup header reading "Trigger ready" or "N issues". new
  • Testing. Classic: leave editor → open runtime Trigger Debugger → select a token → perform action. New: collapsed Test Trigger strip below the mechanical view with a Run Test button that expands the tester and runs with defaults. Author tweaks inputs in the panel and re-runs. Runtime Trigger Debugger is unchanged and stays the right tool for live-scene bugs. new
  • Card auto-derivation. Display card fields (Trigger prose, Effect prose, Distance, Target) are derived from the programmatic fields automatically. Authors can type into a Display override field to replace the derived text with custom prose; blank override falls back to derivation. new
  • Plain-language field labels. "Triggers Only When" (form) / "Triggers When" (mechanical view short label) instead of "Filter" or "Condition Formula". new

What's not changing

Keeping these stable means every existing triggered ability in the compendium keeps working with zero conversion.

  • Data model. TriggeredAbility object, conditionFormula field, trigger id, subject, behaviours list. No schema migration. unchanged
  • Trigger event dispatch. The engine raises triggers the same way. The picker list, subject resolution, and behaviour execution are untouched. unchanged
  • GoblinScript language. No new operators, no syntax changes. Every existing formula keeps parsing and evaluating identically. unchanged
  • Runtime Trigger Debugger. Still available, still the authoritative tool for "a live scene did something unexpected". unchanged

Opt-in technical additions

Four optional pieces make the new editor's affordances work well. None is a blocker: the editor loads and operates without any of them. Each one adds readability or static validation; skipping one degrades a specific feature but never breaks anything.

1. Symbol prose templates opt-in

What: add an optional prose field to each GoblinScript symbol declaration (on RegisterTrigger / RegisterSymbol / inline symbol tables).

damage = {
    name = "Damage",
    type = "number",
    desc = "The amount of damage taken when triggering this event.",
    prose = "the damage",   -- NEW, optional
}

Why: powers the auto-derived card text and the mechanical view's "Triggers when: X" dry-run sentence. A condition like damage >= 3 renders as "the damage is at least 3" instead of raw formula.

If skipped: the card's Trigger row and the dry-run sentence show the raw formula wrapped as "Triggers when: damage >= 3 is true". Mechanical view chips still work. Test Trigger still works. Only the plain-language rendering degrades.

Effort: ~1 day one-time backfill for the ~30-40 existing symbols. ~1 minute per new symbol going forward.

2. Function prose templates opt-in

What: same pattern for GoblinScript function registrations (Distance, Count Nearby Enemies, Stacks, Line of Sight, Friends, etc.). Template uses positional placeholders for arguments.

RegisterSymbol("Distance", {
    fn = function(ctx, target) ... end,
    type = "number",
    prose = "the distance to {1}",   -- NEW, optional. {1} = first argument
})

Why: function-call prose is what makes Count Nearby Enemies(2) >= 3 read as "at least 3 enemies are within 2 squares". Without this, the dry-run shows the raw call verbatim.

If skipped: function calls inside conditions show raw. Same degradation profile as symbols.

Effort: ~15-20 GoblinScript functions to cover. ~1 minute per new function.

3. Subject-event compatibility metadata opt-in

Terminology note: this item refers to the Triggered Ability Subject field (the form dropdown - values like self, attacker, any). Not to be confused with the GoblinScript Subject symbol (a variable referenceable inside condition formulas). The codebase currently calls both "subject" - resolving the ambiguity in the UI is a separate open question.

What: a declarative table of which Subject field values are valid for each registered trigger event. Example: the move event has no attacker concept; setting Subject to attacker on a move trigger silently no-ops at runtime.

Why: powers the "Subject does not match event" warning chip in the mechanical view. Surfaces the mismatch at authoring time instead of discovery-in-scene.

If skipped: no chip. Author finds out at runtime when the trigger silently does nothing and the Trigger Debugger log shows "attacker not available" (TriggeredAbility.lua lines 947-959). Annoying, not broken.

Effort: ~half day. The mapping already exists implicitly in the subject resolver code; it just needs to be extracted into a declarative lookup.

4. Condition prose rendering engine opt-in

What: a new Lua subsystem (probably DMHub Utils/GoblinScriptProse.lua) that takes a compiled GoblinScript formula and returns a plain-English sentence. Uses the symbol and function prose templates above, plus pattern matchers for operator shapes.

Approach: regex/pattern-match the common ~15-20 condition shapes (simple comparisons, boolean chains, has/is, Distance, Count Nearby Enemies, Stacks). Fall back to raw formula for anything else. Covers ~75% of real conditions cleanly in the compendium; the remaining ~25% show raw formula as "Triggers when: [raw formula]".

If skipped: the card's auto-derived Trigger row and the mechanical view's dry-run line always show raw formulas. The mechanical view chips and the Test Trigger simulation still work - those don't depend on prose.

Effort: ~3 days initial implementation. Per new condition shape going forward, optional (degrades gracefully if not added).

What breaks if we adopt none of the opt-in items

Nothing breaks. The editor ships and is usable on day one with only the UI changes. Every opt-in item affects readability or static validation affordances; none affects runtime correctness or data fidelity. The trade-off table below summarises what each skip costs.

Function prose templates Function calls inside conditions show raw (e.g. Count Nearby Enemies(2)) in dry-run. Cosmetic
Subject-event compat metadata No chip when subject doesn't match the event. Author finds out via runtime log. Minor annoyance

Test Trigger panel Author still uses the runtime Trigger Debugger as today. Tighter iteration loop lost but nothing is blocked. Workflow regression

Authoring workflow summary

Side-by-side of how a triggered ability gets made before and after the new editor.

Classic

  1. Open ability in compendium editor.
  2. Scroll long form; fill fields (trigger event, subject, condition formula, behaviours, description).
  3. Save, close, switch to game scene.
  4. Select token, use Trigger Debugger to perform the action, scan handled/unhandled list.
  5. If wrong: return to editor, re-scroll, fix, repeat from step 3.

New

  1. Open ability in editor. Four sections in a left nav - start in Trigger.
  2. Fill fields. Mechanical view to the right updates live: chips go from warn to valid as you fix things. Rollup reads "Trigger ready" when all green.
  3. Click Run Test on the collapsed strip below the mechanical view. Panel expands, defaults run, result shows pass/fail with the concrete values.
  4. Tweak inputs in the panel, re-run. Close the panel when satisfied; the strip remembers the last result.
  5. Move to Display section if you want to polish the player-facing card; most fields auto-derive and don't need touching.
  6. Save. Runtime Trigger Debugger is still available for anything that only reproduces in a live scene.

Definition-of-done gate for new GoblinScript content

When adding a new GoblinScript symbol, function, or trigger event going forward, the team has a choice:

  • Zero-overhead path. Register the symbol/function as today. Skip the prose field. Editor falls back to raw formula rendering for any condition using that symbol. The content works.
  • Polished path. Register the symbol/function, plus add a prose field with a short substitutable phrase. ~1 minute extra. Every condition using that symbol now renders as plain English in the card and dry-run.

Recommendation: treat prose as a soft standard on PR review. Mergeable without it, but reviewer should ask "could you add a prose hint?" as a polite nudge. This keeps the backfill from accumulating while not blocking anyone's work.

Open questions to resolve before Lua implementation

  1. "Pick a replacement" / "Open event picker" action links in the mechanical view's error diagnostics. UX and wire-up still to design (parked by user request).
  2. Pretend token templates for the Test Trigger's attacker / target creature inputs. Stock set (generic enemy / ally / minion / boss) vs. free-form author-defined. Stock is faster; free-form is more flexible.
  3. Test scenario persistence. Should "last run" state survive a close-and-reopen of the editor, or reset on every open? Simpler to reset; saved scenarios could come later if demand shows up.
  4. Deprecation story for TriggeredAbilityDisplay modifier. The classic modifier-based card display still exists in content. New editor derives the card from fields directly. Need a migration plan for content that uses the explicit modifier.
