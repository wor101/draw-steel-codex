About the Test Trigger strip. By default Test Trigger is a single 32px-high strip below the mechanical view - just a title, the last run's summary, and an Open tester button. This keeps the three-pane editor compact on low-resolution displays: the card and mechanical view fit without scroll, and the strip contributes a one-line rollup. The three strip variants shown above are the real states: no test run yet (muted), last run passed (green badge + summary), last run failed (amber badge + reason). Click Open tester to expand the full form - see View 5 for what expanded looks like.
Why collapsed by default, not a modal: the iteration loop is the whole value - tweak the condition, run, read the pass/fail reason, tweak again. A modal breaks that loop (open, run, close, edit, reopen, run). Inline collapsed gives the author a one-click open to the tester that stays in place, keeps the mechanical view visible, and leaves the condition field editable while the tester is open. The strip itself also carries the most-recent outcome as an ambient signal, so the author doesn't have to open the tester to see "did my last test pass?".
What Test Trigger does (when expanded):
  • Generates a form with one input per symbol the condition references. For the trigger event's declared symbols you get a typed control (number input, text dropdown, boolean toggle, token picker, keyword chips). For map-state functions (Count Nearby Enemies, Distance, Line of Sight) you get a "pretend value" input.
  • Compiles the condition and evaluates it against your inputs when you press Run. Shows pass or fail, and the concrete sub-expression values that decided the outcome (3 >= 3 = true).
  • Highlights which sub-condition failed when compound conditions don't pass, so A and B and C tells you "B was false".
  • Previews the behaviour list that would execute on pass (names and key parameters - not execution).
  • Leaves the most-recent outcome visible on the collapsed strip after close.
What it does not do:
  • Execute behaviours for real. No damage is applied, no conditions inflicted, no resources spent, no tokens moved.
  • Model map geometry. Adjacency, line of sight, cover, terrain - those are "pretend" inputs. If you want to verify an actual scene, that's still the runtime Trigger Debugger's job.
  • Simulate multi-trigger cascades. If triggering this ability would raise another event (e.g. a damage behaviour triggering a damage-taken reaction), that chain isn't followed.
  • Replace the runtime Trigger Debugger. When a trigger misbehaves live in a real scene - wrong target, unexpected fail, double-fire - the runtime debugger has the actual game state to inspect. Test Trigger is for authoring-time sanity.
  • Persist test scenarios across sessions (initially). If saved scenarios prove useful, we can add a "pin this test case" affordance later.
When to reach for which tool: while authoring a new or changed trigger, Test Trigger is the tighter loop - type a condition, pick inputs, run, read the pass/fail reason, adjust. When something surprises you during a real session, the runtime Trigger Debugger is the right answer because it has the live scene state Test Trigger can't see.

Design rules for the mechanical view:
  1. Chip > colour. Every problem state has a labelled chip. Colour is reinforcement, not the signal.
  2. Rollup on top. Author sees "Trigger ready" / "N issues" / "N notes" without scanning every row.
  3. Quiet on happy path. No chips when everything is valid. Avoids the "wall of green checkmarks" dead space that teaches users to ignore the pane.
  4. Silent is a legitimate mode. When silent: true is set, empty-behaviours and blank-description chips are suppressed entirely because the intent is bookkeeping, not effect (see cell 6).
  5. Dry-run line at the bottom - one plain-English sentence: "Triggers when: X" or "Triggers when: never (reason)". This is the final honesty check and catches semantic issues even when the syntax is valid.
  6. Diagnostics list explains why - each issue says what broke and, where possible, how to fix it. Never just a colour.
  7. Accessible by default. Chip text + rollup text carry the signal without colour, so colourblind users and screen readers get the same information.
What we chose NOT to flag (and why):
  • Unregistered trigger event ids. Real trigger renames are rare; the earlier "100+ occurrences" crawl was wrong (it missed MCDMRules.lua registrations). Worth a one-time migration report, not a per-ability chip.
  • "Condition is always true" / "always false" by static analysis. Requires a GoblinScript AST we don't have. The Test Trigger panel (View 3) covers this case - the author runs a simulation and sees the evaluation result.
  • Conflicting when-active constraints. Only one real example in the bestiary and it's fuzzy to diagnose. Park until we see it in the wild.

Design decisions visible in these variants:
  • Typed controls, not raw inputs. Each symbol type maps to the right UI - slider for numbers, dropdown for text enums, chip row for sets, toggle for booleans, token picker for creature references. The author never types a number into a "damage" text box.
  • Origin line under each label. The small grey line below each input label ("symbol: damage (number)", "pretend (used by Count Nearby Enemies)") tells the author where the input comes from and whether it's real event data or a substituted map-state value.
  • Inputs referenced by the condition are foregrounded; inputs that aren't used are dimmed. See Variant C: the condition only uses damage and damagetype, so the other three inputs collapse to a one-line summary. Saves vertical space and tells the author "changing these wouldn't change the outcome".
  • Pretend tokens are first-class. The attacker input in Variant B shows "Pretend Goblin" - a stock template with a keyword set and a level. Authors can swap to a real scene token, a stock template ("generic enemy", "generic ally", "generic minion"), or back to pretend. This matters for conditions that do keyword checks on the attacker (Attacker.Keywords has "Melee").
  • Failure diagnostics call out the first falsy sub-condition. In a long A and B and C chain, knowing which link broke is the answer the author actually needs. Variant C shows this: "First failing sub-condition: damage >= 3. Required damage of at least 3, got 2."
  • Pass result previews behaviour list with substituted values. Variant B's result lists "Deal 3 damage to the attacker (Pretend Goblin)" - substituting the actual attacker name into the behaviour preview. Makes the simulation feel concrete.
What these variants still don't do (carried over from View 3's list, repeated here so the scope of the tool stays honest):
  • No real behaviour execution - the "Pretend Goblin" takes no damage.
  • No map geometry - adjacency, line of sight, cover, terrain are all "pretend" inputs.
  • No multi-trigger cascades - if the 3 damage would raise another losehitpoints event on the pretend goblin and trigger its reactions, that chain isn't followed.
  • No scenario persistence yet - close the editor and the test inputs reset. Adding a "pin this test case" option can come later if demand shows up.
